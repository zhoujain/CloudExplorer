# 04 · 后端核心机制

> 本章聚焦后端最值得学习的四块：鉴权、权限、菜单、定时任务。代码主要在 `framework/sdk/backend` 与各模块的 `constants` 包。

## 1. 鉴权：JWT + 自定义 Header

### Token 流转

1. 用户登录成功，后端签发 JWT，放入响应 header `CE-TOKEN`。
2. 前端把 `CE-TOKEN` 存起来，后续请求带上。
3. 同时带上 `CE-ROLE`（角色）和 `CE-SOURCE`（工作空间 id）。

### 多租户上下文

后端从 header 解析出 `{用户, 角色, 工作空间}` 三元组，注入到当前请求上下文。如果角色/工作空间与用户实际不符，降级为 `ANONYMOUS`。这套机制让"同一个用户切换工作空间"只需前端改 header，无需重新登录。

> 学习建议：在 `sdk/backend` 中找 JWT 过滤器/拦截器（搜索 `CE-TOKEN`、`cepc`、`CeSecurityExpressionRoot`），通读一遍 token 解析与上下文构建流程。

## 2. 权限模型

### 声明权限

每个模块在 `PermissionConstants.java` 用 Builder 链式声明：

```java
private static final ModulePermission.Builder MODULE_PERMISSION_BUILDER =
    new ModulePermission.Builder()
        .group(new PermissionGroup.Builder()
            .id(GROUP.USER)              // 权限组
            .name("i18n_permission_user")
            .permission(new Permission.Builder()
                .operate(OPERATE.READ)   // 操作：READ/CREATE/EDIT/DELETE...
                .name("i18n_permission_user_read")
                .role(RoleConstants.ROLE.ADMIN)
                .role(RoleConstants.ROLE.ORGADMIN))
            .permission(new Permission.Builder()
                .require(OPERATE.READ)   // 依赖 READ 才能授予 CREATE
                .operate(OPERATE.CREATE)
                .name("i18n_permission_user_create")
                .role(RoleConstants.ROLE.ADMIN)
                .role(RoleConstants.ROLE.ORGADMIN))
        );
```

要点：
- **权限 = 权限组 + 操作**，如 `USER:READ`。
- `require` 表达权限依赖（授 CREATE 必须先有 READ）。
- `role` 限定该权限对哪些角色可见/可授予。

### 在接口上鉴权

```java
// 推荐：当前模块内判断
@PreAuthorize("@cepc.hasCePermission('USER:READ')")
public void someMethod() { ... }

// 多权限任一命中
@PreAuthorize("@cepc.hasAnyCePermission('USER:READ','USER:CREATE')")
public void someMethod() { ... }

// 跨模块/原生写法：需带模块名前缀
@PreAuthorize("@cepc.hasAuthority('[management-center]USER:READ+CREATE')")
public void someMethod() { ... }
```

`cepc` 是自定义的权限表达式 Bean，实现见 `CeSecurityExpressionRoot.java`。它根据当前上下文的角色与工作空间，判断用户是否拥有目标权限。

> 学习建议：对比 Spring Security 原生的 `hasAuthority`，体会 CE-Lite 为什么封装一层 `hasCePermission`——因为它要处理"模块名 + 权限组 + 操作 + 多租户角色"四维判断。

## 3. 菜单与路由

`MenuConstants.java` 的 `MENUS_BUILDER` 用 Builder 声明菜单树。每个菜单项包含：

- `name`：菜单唯一 id
- `path` / `componentPath`：路由路径与对应 Vue 组件
- `requiredPermission`：访问该菜单需要的权限（角色 + 权限）
- `childMenu` / `childOperationRoute`：子菜单 / 子操作路由
- `order`：排序；`quickAccess`/`saveRecent`：首页快捷入口与最近访问

两种典型形态：
- **单级菜单**：菜单下挂操作路由（如列表页内的"创建"路由），通过 `childOperationRoute` 配置，不出现在左侧菜单但在 URL 可达。
- **层级菜单**：父菜单 + `childMenu` 子菜单。

菜单声明会序列化下发给前端，前端据此动态生成路由与左侧导航——这是"声明式扩展"在前端的体现。

## 4. 定时任务

`JobConstants.java` 用 `JobSetting` 声明任务，支持两类触发：

- **cron 表达式**：`"0 0 0 * * ? *"`
- **时间间隔**：`25` + `DateBuilder.IntervalUnit.MINUTE`

任务执行器需集成 `AsyncJob` 并实现 `Job` 接口。

任务组决定调度语义：

| 任务组 | 含义 |
|----|----|
| `SYSTEM_GROUP` | 系统级任务，启动即创建一个 |
| `CLOUD_ACCOUNT_RESOURCE_SYNC_GROUP` | 每个云账号自动分配一个，可在管理中心改时间 |
| `CLOUD_ACCOUNT_BILL_SYNC_GROUP` | 账单模块自定义，按云账号分配 |
| `CLOUD_COMPLIANCE_RESOURCE_SYNC_GROUP` | 安全合规模块自定义 |

`JobSetting` 里几个 `platform -> boolean` 的 lambda 控制：该任务在某个云平台是否展示、是否允许开关、是否允许改时间。这是"按云平台差异化配置任务"的函数式写法，值得借鉴。

## 5. 数据访问与迁移

- **MyBatis-Plus**：实体映射 + 条件构造器 + 分页，代码里大量 `BaseMapper`、`IService`。
- **Flyway**：`resources/db/migration/V*.sql`，启动自动执行。改表结构 = 新增一个版本脚本，不要改历史脚本。
- **Redisson**：分布式锁（如同步任务防重入）、缓存。

## 6. 国际化

`resources/i18n/` 放多语言文案，权限/菜单 name 都用 `i18n_xxx` key 引用，前端按当前语言渲染。新增功能时记得同步补齐中/英/繁体三套文案。

## 7. 云厂商对接（provider）

`framework/provider` 定义统一的云资源操作接口，各云厂商实现。业务模块（vm-service）调用抽象接口，不关心具体是阿里云还是 VMware——这是经典的**策略/适配器模式**在云管领域的落地。
