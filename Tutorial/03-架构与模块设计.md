# 03 · 架构与模块设计

## 1. 整体架构

CE-Lite 是**微服务 + 前端微前端化**架构：

```
                        ┌──────────────┐
   浏览器 ────────────► │   Gateway    │  统一入口、路由、鉴权前置
                        └──────┬───────┘
                               │
            ┌──────────────────┼──────────────────┐
            ▼                  ▼                  ▼
   ┌────────────────┐  ┌───────────────┐  ┌──────────────────┐
   │ management-    │  │  vm-service   │  │ finance/operation│
   │ center (基座)  │  │  (业务模块)   │  │ /security        │
   └───────┬────────┘  └───────┬───────┘  └────────┬─────────┘
           │                   │                   │
           └─────────┬─────────┴───────────────────┘
                     ▼
              ┌─────────────┐    ┌─────────┐
              │   Eureka    │    │ MySQL   │
              │  (注册中心) │    │ Redis   │
              └─────────────┘    └─────────┘
```

- **Gateway**：所有外部请求入口，根据路径路由到对应微服务。
- **Eureka**：服务注册发现，模块启动后注册自己，Gateway 据此转发。
- **management-center**：基座，提供用户/角色/组织/工作空间/云账号/定时任务管理等"平台级"能力。
- **业务模块**：vm-service、finance-management、operation-analysis、security-compliance，各自独立部署、独立数据库表。

## 2. framework（基座）各模块职责

| 子模块 | 职责 |
|----|----|
| `eureka` | 服务注册中心 |
| `gateway` | API 网关，路由 + 鉴权 |
| `management-center` | 平台管理：用户、组织、工作空间、云账号、定时任务、系统设置 |
| `sdk` | 前后端公共 SDK：封装鉴权、权限表达式、菜单/权限/任务常量基类、公共前端组件（`commons`） |
| `provider` | 云厂商对接抽象层：定义统一接口，各云厂商实现资源同步/操作 |

`sdk` 是理解整个项目"约定"的钥匙——所有模块都依赖它复用基座能力。前端 `sdk/frontend/commons` 提供了跨模块复用的 stores、组件、工具函数。

## 3. services（业务模块）

每个业务模块都是 **backend + frontend** 一体的自治单元：

- **vm-service**：云主机管理（开/关/停/重启、迁移、控制台等）
- **finance-management**：云账单采集与展示
- **operation-analysis**：运营分析报表
- **security-compliance**：安全合规扫描

模块之间通过 **HTTP（经 Gateway）** 交互，而非直接数据库共享——这是微服务的标准纪律。

## 4. 模块的"三大声明"

每个模块通过三个常量类向平台**声明式注册**自己的扩展点：

| 常量类 | 作用 | 位置 |
|----|----|----|
| `PermissionConstants.java` | 声明模块的权限组与权限项 | `backend/.../constants/` |
| `MenuConstants.java` | 声明菜单树与路由 | 同上 |
| `JobConstants.java` | 声明定时任务 | 同上 |

平台启动时会收集各模块的声明，统一构建权限表、菜单树、任务调度。这是 CE-Lite 模块化的核心机制——**加功能 = 加模块 + 声明三个常量**。

## 5. 多租户模型

角色三级：

- `ADMIN`：系统管理员（不传 `CE-SOURCE`）
- `ORGADMIN`：组织管理员
- `USER`：普通用户

资源通过 **组织 / 工作空间** 隔离，`CE-SOURCE` 携带当前操作的工作空间 id。云账号、云主机等资源都关联到具体工作空间，实现多租户数据隔离。

## 6. 前端多包结构

根 `package.json` 用 Yarn workspaces 把 `framework/**` 和 `services/**` 下所有前端子包纳入一个 monorepo：

- **基座前端（ce-base / sdk frontend）**：登录、布局、菜单、首页、公共组件。
- **各业务前端**：自己的 views/api/stores，构建产物由基座运行时按路由加载。

这种"基座 + 子模块前端"的模式类似一种轻量微前端，避免把所有页面塞进一个巨型应用。

## 7. 数据库与迁移

- 每个模块**独立建表**，互不干扰。
- 用 **Flyway** 管理迁移脚本，放在 `backend/src/main/resources/db/migration/`。
- 学习时建议先看 `vm-service` 的迁移脚本，能快速理解核心数据模型（云账号、云主机、同步记录等）。
