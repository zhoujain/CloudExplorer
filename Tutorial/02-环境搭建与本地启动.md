# 02 · 环境搭建与本地启动

> 本章对应官方 `doc/开发指南.md` 的"环境准备 / 开发"部分，补充一些理解性说明。

## 1. 准备依赖软件

### 后端

- **JDK 17**（Spring Boot 3.1 强制要求 17+）
- **Maven**（仓库自带 `mvnw`/`mvnw.cmd`，可直接用包装器）
- **MySQL**：用于各模块业务数据
- **Redis**：缓存、分布式锁、token 等
- **Docker**：用于打包/运行容器化部署（本地开发可不强制）

### 前端

- **Node.js**（>=16.10 推荐）
- **Corepack**：用于启用 Yarn 3
  ```bash
  corepack enable          # Node >=16.10
  # 或 npm i -g corepack   # Node <16.10
  ```

## 2. 一次性初始化

```bash
# 1) 安装 VMware 相关依赖到本地仓库（根目录执行，只需一次）
mvn initialize

# 2) 安装前端依赖（Yarn workspaces 会一次性装好所有子包依赖）
yarn install
```

> 为什么需要 `mvn initialize`？项目对接 VMware vSphere 需要一些不在公共 Maven 仓库的 SDK 依赖，`initialize` 阶段会把它们装入本地仓库。

## 3. 后端启动顺序

微服务架构，启动有依赖关系。建议顺序：

1. **eureka**（服务注册中心）——其他服务都向它注册
2. **gateway**（网关）——统一入口，路由到各模块
3. **management-center**（管理中心）——基座核心，提供登录/用户/云账号
4. **vm-service / finance-management / operation-analysis / security-compliance** —— 按你要调试的功能按需启动

> 实操提示：先看 `framework/eureka` 的 `application.yml` 确认端口；各模块启动前确保 MySQL/Redis 已就绪，并在对应 `application.yml`（或外部 `cloudexplorer.properties`）中配置好连接信息。

## 4. 前端启动

根 `package.json` 已为每个模块暴露了快捷脚本：

```bash
yarn dev:base        # 启动基座（ce-base，sdk 的前端）
yarn dev:manage      # management-center
yarn dev:vm          # vm-service
yarn dev:bill        # finance-management
yarn dev:security    # security-compliance
yarn dev:operation   # operation-analysis
```

开发时通常**同时启动基座 + 你要调试的模块前端**：基座是外壳（登录、菜单、布局），子模块页面通过基座加载。可以用两个终端分别跑：

```bash
# 终端 A
yarn dev:base
# 终端 B
yarn dev:vm
```

也可以一条命令并行起所有：`yarn dev`（资源占用大，按需使用）。

## 5. 接口调用约定（重要）

调用任意需认证接口，响应 header 会带回当前用户的 JWT token：`CE-TOKEN`。

后续请求需带以下 header：

| Header | 含义 |
|----|----|
| `CE-TOKEN` | 登录拿到的 JWT |
| `CE-ROLE` | 角色：`ADMIN` / `ORGADMIN` / `USER` |
| `CE-SOURCE` | 组织/工作空间 id（`ADMIN` 角色不传） |

若 `CE-ROLE` / `CE-SOURCE` 与当前用户不匹配，后端视为 `ANONYMOUS`。

> 学习要点：这套自定义 header 设计是理解其多租户鉴权的钥匙，见 [04 · 后端核心机制](./04-后端核心机制.md)。

## 6. 常见踩坑

- **JDK 版本不对**：用 Java 8/11 启动 Spring Boot 3.1 会直接报错，务必 17+。
- **Yarn 版本**：项目用 Yarn 3.5.1（Berry），与 Yarn 1 行为不同，依赖 `.yarnrc.yml`，不要手动改成 npm。
- **MySQL 字符集/时区**：建议 utf8mb4，时区与后端一致，避免 Flyway 迁移或账单时间错乱。
- **模块前端端口冲突**：每个模块前端有独立 dev 端口（见各自 `vite.config.ts`），新模块端口在脚手架里指定，避免占用。
